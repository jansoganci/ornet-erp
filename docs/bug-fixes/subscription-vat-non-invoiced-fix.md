# Bug Fix: Incorrect VAT Calculation for Non-Invoiced Subscriptions

**Date:** 2026-04-02  
**Migration:** `00181_fix_non_invoiced_subscription_vat.sql`  
**Status:** ✅ Fixed

---

## Problem Description

### Expected Behavior
- When a subscription has `official_invoice = TRUE`: Add VAT (default 20%) to the base price
- When a subscription has `official_invoice = FALSE`: No VAT should be added. Total payment amount should equal the base price (NET amount)

### Actual Bug
The system was adding VAT to **ALL** subscriptions, regardless of the `official_invoice` flag. This caused non-invoiced subscriptions to have incorrect payment amounts.

---

## Root Cause Analysis

### Database Schema
1. **`subscriptions` table** has an `official_invoice` boolean column (added in migration `00022_subscription_target_fields.sql`)
   - Default value: `true`
   - Purpose: Indicates whether the subscription should generate invoiced payments

2. **`subscription_payments` table** has a `should_invoice` boolean column (added in migration `00018_invoice_logic.sql`)
   - Default value: `true`
   - Purpose: Per-payment flag indicating if an invoice should be issued

### Bug Location
Two functions were generating payments without respecting the `official_invoice` flag:

1. **`generate_subscription_payments()`** (lines 36-37 in `00148_fix_payment_start_from_current_date.sql`)
   ```sql
   v_vat := ROUND(v_subtotal * COALESCE(v_sub.vat_rate, 20) / 100, 2);
   ```
   - Always calculated VAT regardless of `official_invoice`
   - Used when creating/editing subscriptions

2. **`extend_active_subscription_payments()`** (lines 89-90 in `00110_rolling_subscription_payments.sql`)
   ```sql
   v_vat := ROUND(v_subtotal * COALESCE(v_sub.vat_rate, 20) / 100, 2);
   ```
   - Always calculated VAT regardless of `official_invoice`
   - Used by monthly scheduler to generate future payments

---

## Solution

### Changes Made

#### 1. Fixed `generate_subscription_payments()`
**Before:**
```sql
v_vat := ROUND(v_subtotal * COALESCE(v_sub.vat_rate, 20) / 100, 2);
```

**After:**
```sql
-- FIX: Only calculate VAT if official_invoice is true
IF COALESCE(v_sub.official_invoice, true) THEN
  v_vat := ROUND(v_subtotal * COALESCE(v_sub.vat_rate, 20) / 100, 2);
ELSE
  v_vat := 0;
END IF;
```

Also added `should_invoice` to the INSERT statement:
```sql
INSERT INTO subscription_payments (
  subscription_id,
  payment_month,
  amount,
  vat_amount,
  total_amount,
  should_invoice  -- NEW: Set based on subscription's official_invoice
)
VALUES (
  p_subscription_id,
  v_month,
  v_subtotal * v_multiplier,
  v_vat     * v_multiplier,
  v_total   * v_multiplier,
  COALESCE(v_sub.official_invoice, true)  -- NEW
)
```

#### 2. Fixed `extend_active_subscription_payments()`
Applied the same logic as above:
- Check `official_invoice` before calculating VAT
- Set `should_invoice` flag on generated payments
- Updated subtotal calculation to include all fee components (`static_ip_fee`, `sim_amount`)

## Data Migration Considerations

### Existing Payments
This migration **does NOT** retroactively fix existing payment records. It only affects:
1. New payments generated after the migration
2. Future payments generated by the monthly scheduler

### If Retroactive Fix is Needed
If you need to fix existing incorrect payments, run this query:

```sql
-- WARNING: Test on staging first!
-- This updates existing payments for non-invoiced subscriptions

UPDATE subscription_payments sp
SET 
  vat_amount = 0,
  total_amount = amount,
  should_invoice = false
FROM subscriptions s
WHERE sp.subscription_id = s.id
  AND s.official_invoice = false
  AND sp.status = 'pending'  -- Only update unpaid payments
  AND sp.vat_amount > 0;     -- Only update those with VAT
```

## Finance Impact

### Before Fix
- Non-invoiced subscriptions were generating payments with 20% VAT
- This inflated revenue numbers in financial reports
- Customers were being charged more than intended

### After Fix
- Non-invoiced subscriptions generate payments with 0% VAT
- `total_amount = amount` (NET amount only)
- Financial reports will show correct revenue
- `should_invoice` flag is properly set for downstream processing

### Reporting Considerations
- The `financial_transactions` table is the single source of truth for financial reporting
- When subscription payments are marked as paid, they create income rows in `financial_transactions`
- The `should_invoice` flag determines whether `output_vat` is recorded
- See `fn_subscription_payment_to_finance()` trigger in migration `00050_subscription_payment_to_finance.sql`

## Architecture Notes

### Finance Architecture Rules (from CLAUDE.md)
1. **`financial_transactions` is the single source of truth** — All financial reporting reads from this table
2. **Always use dynamic `vat_rate`** — Never hardcode VAT percentages
3. **Amount fields are always NET (KDV haric)** — VAT is calculated and stored separately
4. **Respect the `should_invoice` flag** — This determines whether VAT is recorded in finance

### VAT Calculation Flow
```
Subscription Creation
  ↓
generate_subscription_payments()
  ↓ (checks official_invoice)
subscription_payments (with vat_amount, should_invoice)
  ↓ (when marked paid)
fn_subscription_payment_to_finance() trigger
  ↓ (checks should_invoice)
financial_transactions (with output_vat)
  ↓
Financial Reports (P&L, VAT, Income)
```
